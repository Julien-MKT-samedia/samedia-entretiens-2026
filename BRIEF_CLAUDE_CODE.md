## FONCTIONNALITÉS REQUISES

La plateforme doit proposer **4 modules** accessibles pour chaque candidat :

### Module 1 — Fiche synthèse candidat
Pour chaque candidat, une fiche structurée pré-remplie contenant :
- Informations clés (nom, âge, localisation, formation, situation)
- Résumé de parcours en 3-5 points clés
- Points forts vs le poste SAMEDIA (3-5 arguments)
- Points de vigilance / questions à creuser (3-5 points)
- Score de matching initial (sur 10, pré-calculé par toi selon l'analyse)

### Module 2 — Questions d'entretien personnalisées
Pour chaque candidat, une liste de questions d'entretien structurées en catégories :
- **Motivation / Reconversion** (pourquoi SAMEDIA, pourquoi maintenant)
- **Expérience commerciale** (méthodes, outils, résultats chiffrés)
- **Connaissance technique BTP/outillage**
- **Gestion de secteur** (organisation, prospection, reporting)
- **Softs skills / Fit culture**
- Questions spécifiques au profil (tirées des points de vigilance)

Chaque question doit être affichée avec un champ de prise de notes associé.

### Module 3 — Grille de scoring interactive
Critères d'évaluation pondérés adaptés au poste SAMEDIA :
1. **Expérience TC terrain** (coefficient 3) — /5
2. **Connaissance réseaux négoce BTP** (coefficient 3) — /5
3. **Couverture secteur géo** (coefficient 2) — /5
4. **Technicité BTP / outillage** (coefficient 2) — /5
5. **Autonomie / Organisation** (coefficient 2) — /5
6. **Profil commercial / Closing** (coefficient 2) — /5
7. **Stabilité / Sérieux du parcours** (coefficient 1) — /5
8. **Motivation exprimée** (coefficient 1) — /5

Le score est calculé automatiquement en temps réel au fur et à mesure que Julien saisit les notes pendant l'entretien. Les notes et scores sont sauvegardés en **Supabase** (voir section technique ci-dessous).

### Module 4 — Tableau comparatif final
Vue synthétique de tous les candidats avec :
- Classement par score total (dynamique)
- Barre de progression visuelle par critère
- Champ "Décision" par candidat (À convoquer / Convoqué / Retenu / Non retenu)
- Export ou impression possible

## SPÉCIFICATIONS TECHNIQUES

### Structure des fichiers
- `index.html` — fichier principal (application SPA complète)
- `data/candidates.js` — toutes les données candidats (si séparé)
- `style.css` — styles (si séparé)

Privilégier un fichier unique `index.html` si possible pour simplifier le déploiement Netlify. Si plusieurs fichiers, tous doivent être référencés correctement.

### Technologies
- HTML5 / CSS3 / JavaScript vanilla (pas de framework obligatoire)
- Tailwind CSS via CDN acceptable
- Pas de build tool, pas de serveur custom
- **Persistance via Supabase** (base de données PostgreSQL managée, SDK JS via CDN)
- Responsive (utilisable sur laptop)

### Supabase — Architecture multi-utilisateurs

L'application est utilisée par **plusieurs recruteurs en simultané** (au moins 2). Chaque utilisateur saisit ses propres scores et notes, et peut **voir les commentaires des autres** en temps réel ou au rechargement. Les données sont partagées via Supabase, sans authentification complexe : une simple sélection de prénom au démarrage suffit.

Utiliser le client JS Supabase via CDN : `https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2`

#### Identification utilisateur (sans auth Supabase)
Au premier lancement, afficher une **modale de sélection d'utilisateur** simple :
```
Qui êtes-vous ?
[ Julien ]   [ + Autre prénom... ]
```
Le prénom choisi est stocké en `sessionStorage` (pas Supabase) et préfixe toutes les écritures. Pas de mot de passe — c'est un outil interne.

#### Tables à créer

```sql
-- Scores par candidat, par critère, PAR UTILISATEUR
CREATE TABLE interview_scores (
  id uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  user_name text NOT NULL,            -- ex: 'Julien', 'Marie'
  candidate_id text NOT NULL,         -- ex: 'guilbault', 'charrier'
  criterion_key text NOT NULL,        -- ex: 'exp_tc', 'reseau_btp'
  score integer CHECK (score BETWEEN 0 AND 5),
  updated_at timestamptz DEFAULT now(),
  UNIQUE (user_name, candidate_id, criterion_key)
);

-- Notes libres par candidat, par question, PAR UTILISATEUR
CREATE TABLE interview_notes (
  id uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  user_name text NOT NULL,
  candidate_id text NOT NULL,
  question_key text NOT NULL,         -- identifiant unique de la question
  note_text text,
  updated_at timestamptz DEFAULT now(),
  UNIQUE (user_name, candidate_id, question_key)
);

-- Décision finale par candidat, PAR UTILISATEUR
CREATE TABLE interview_decisions (
  id uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  user_name text NOT NULL,
  candidate_id text NOT NULL,
  decision text,                      -- 'À convoquer' | 'Convoqué' | 'Retenu' | 'Non retenu'
  global_comment text,
  updated_at timestamptz DEFAULT now(),
  UNIQUE (user_name, candidate_id)
);
```

#### Stratégie de lecture/écriture multi-utilisateurs

- **Écriture :** upsert sur la contrainte UNIQUE `(user_name, candidate_id, criterion_key)` — chaque utilisateur écrase uniquement ses propres données
- **Lecture :** au chargement de chaque fiche candidat, fetcher **toutes les lignes** pour ce `candidate_id` (tous utilisateurs confondus), puis afficher :
  - Les scores/notes de l'utilisateur courant dans les champs éditables
  - Les scores/notes des **autres utilisateurs en lecture seule**, clairement identifiés par leur prénom (ex: bandeau ou colonne "Avis de Marie : ...")
- **Tableau comparatif :** afficher le score moyen toutes personnes confondues + les scores individuels par utilisateur côte à côte
- **Temps réel (optionnel mais recommandé) :** utiliser `supabase.channel().on('postgres_changes', ...)` pour rafraîchir automatiquement les avis des autres sans recharger la page

#### Variables de configuration
```js
const SUPABASE_URL = 'SUPABASE_URL_ICI';
const SUPABASE_ANON_KEY = 'SUPABASE_ANON_KEY_ICI';
```
Julien fournira les valeurs réelles au moment du déploiement.

**RLS (Row Level Security) :** désactiver pour cet usage interne (`ALTER TABLE ... DISABLE ROW LEVEL SECURITY`) ou configurer une policy `anon` full-access. L'app est privée, accessible uniquement via URL Netlify.

- Au démarrage : modale de sélection du prénom utilisateur (stocké en sessionStorage)
- Navigation entre candidats fluide (sidebar ou onglets)
- Navigation entre les 4 modules par candidat (tabs ou sections)
- Style : **minimaliste, outil de travail efficace** — priorité à la lisibilité et à l'utilité sur l'esthétique
- Les données candidats sont pré-remplies en JS ; scores et notes sont saisis pendant l'entretien et sauvegardés en Supabase
- **Affichage multi-utilisateurs :** dans chaque fiche, les avis des autres recruteurs apparaissent clairement distincts (badge prénom + couleur différente), en lecture seule à côté des champs éditables de l'utilisateur courant
- **Tableau comparatif :** score moyen global + scores par utilisateur affichés côte à côte pour chaque candidat
